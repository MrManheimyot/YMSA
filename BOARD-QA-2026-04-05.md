## §2 — HARDCODED / FAKE DATA VIOLATIONS

### CRIT-03: Hardcoded Correlation Matrix — Fake Financial Data 🔴

**File**: `src/broker-manager/flush-cycle.ts` lines 55–65

```
CORRELATION_MATRIX = {
  AAPL: { MSFT: 0.82, GOOGL: 0.78, META: 0.75, AMZN: 0.72, NVDA: 0.70, ... },
  MSFT: { GOOGL: 0.80, META: 0.72, ... },
  NVDA: { AMD: 0.90, AVGO: 0.85, ... },
  ...
}
```

**Impact**: Used in `correlationCheck()` to GATE trades. Stale correlations = wrong blocking decisions.  
**Fix**: Compute rolling 90-day correlations from Yahoo OHLCV data, or use live correlation service.

### CRIT-04: Hardcoded 50-Symbol Universe Fallback 🔴

**File**: `src/cron/morning-briefing.ts` lines 180–189

```
const hardcodedUniverse = ['AAPL', 'MSFT', 'GOOGL', ... /* 50 symbols */];
```

**Impact**: If `TIER1_WATCHLIST` / `TIER2_WATCHLIST` env vars missing, this fallback activates SILENTLY. Masks configuration failures.  
**Fix**: Throw explicit error if env watchlists are empty. Never fall back to hardcoded lists.

### HIGH-02: Default ATR Estimate 🟠

**File**: `src/cron/market-scans.ts` line 382

```
atr: quote.price * 0.02,  // Default estimate when ATR missing
```

**Impact**: 2% price estimate is not real volatility. Can produce misleading stop-loss calculations.  
**Fix**: Require real ATR from `indicators.ts` or skip signal if ATR unavailable.

### MED-02: `IncludeMockTick` API Parameter 🟡

**File**: `src/api/marketwatch.ts` line 100

```
IncludeMockTick: 'true',
```

**Status**: This is a MarketWatch/WSJ API parameter name (not our naming). Likely requests extended-hours tick approximations, NOT fake data. Needs verification against WSJ's API docs.

## §5 — DATABASE INTEGRITY

### HIGH-05: `getPnlDashboardData()` — Full Table Scan 🟠

**File**: `src/db/queries/pnl-queries.ts`

```sql
SELECT * FROM daily_pnl ORDER BY date ASC          -- loads ALL rows
SELECT * FROM trades WHERE status = 'CLOSED'       -- loads ALL rows
SELECT pnl FROM trades WHERE status = 'CLOSED'     -- DUPLICATE scan
```

**Impact**: On mature system with 365+ daily_pnl rows and 1000+ trades, this query set will take 500-800ms per dashboard load.  
**Fix**: Add `WHERE date >= date('now', '-365 days')` and remove duplicate pnl query.
